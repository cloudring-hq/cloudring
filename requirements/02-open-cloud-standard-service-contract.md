# Open Cloud Standard And Service Contract

Open Cloud Standard - это общий язык между CloudRING, сервисами, провайдерами,
private-инсталляциями, marketplace, billing, monitoring и AI-агентами.

Service Connector - механизм, который адаптирует конкретный продукт или сервис
к этому общему языку.

## Требования К Стандарту

| ID | Требование | Почему |
|---|---|---|
| CR-OCS-001 | CloudRING должен иметь формальный Open Cloud Standard, описывающий минимальный контракт сервиса. | Без стандарта невозможны федерация, marketplace, переносимость и автоматизация. |
| CR-OCS-002 | Стандарт должен отделять обязательные capability от опциональных. | Нельзя требовать бесконечно сложный сервис от каждого участника. |
| CR-OCS-003 | Стандарт должен поддерживать регистрацию сервиса и объявление его возможностей. | Платформа должна понимать, что сервис умеет, где доступен и какие условия предлагает. |
| CR-OCS-004 | Стандарт должен описывать lifecycle сервиса: регистрация, проверка, публикация, заказ, provisioning, изменение, остановка, удаление, обновление, миграция. | Marketplace и provider federation требуют одинакового понимания жизненного цикла. |
| CR-OCS-005 | Стандарт должен описывать usage/consumption metrics для биллинга и аналитики. | Экономика платформы невозможна без измеримого потребления ресурсов. |
| CR-OCS-006 | Стандарт должен описывать health, readiness, metrics, logs и traces как обязательные наблюдаемые поверхности. | Платформа должна управлять сервисами, а не угадывать их состояние. |
| CR-OCS-007 | Стандарт должен поддерживать зависимости сервиса от других сервисов. | Реальные продукты состоят из цепочек: база, storage, secret store, queue, identity, network. |
| CR-OCS-008 | Стандарт должен поддерживать несколько сред: local, staging, production, private, edge, disconnected. | Сервис должен проходить один lifecycle в разных контекстах. |
| CR-OCS-009 | Стандарт должен позволять сервису иметь backend и UI, поставляемые независимым разработчиком. | Marketplace должен принимать полноценные продукты, а не только инфраструктурные блоки. |
| CR-OCS-010 | Стандарт должен задавать правила micro-frontend или UI-встраивания без привязки к одному frontend-фреймворку. | UI должен развиваться и подключаться как часть сервиса, но не ломать портал. |
| CR-OCS-011 | Стандарт должен поддерживать machine-readable описание требований к ресурсам: compute, network, storage, identity, secrets, regions, policies. | AI-агенты и scheduler должны принимать решения автоматически. |
| CR-OCS-012 | Стандарт должен поддерживать версионирование и обратную совместимость. | История API показывает, что контракты меняются, а клиенты не могут мигрировать мгновенно. |
| CR-OCS-013 | Стандарт должен иметь deprecated-механизм с явными сроками и заменой. | Старые API нужно выводить из эксплуатации контролируемо. |
| CR-OCS-014 | Стандарт должен поддерживать сертификацию совместимости сервиса. | Пользователь должен понимать, какие сервисы безопасно ставить в public/private/edge. |
| CR-OCS-015 | Стандарт должен поддерживать policy-ограничения: jurisdiction, data residency, forbidden locations, required encryption, required audit. | Compliance должен быть частью выбора сервиса, а не ручной проверкой. |
| CR-OCS-016 | Стандарт должен описывать границы ответственности между CloudRING, провайдером, ISV и клиентом. | В federation-mode инциденты пересекают несколько организаций. |
| CR-OCS-017 | Стандарт должен поддерживать идемпотентные операции lifecycle и usage ingestion. | Повторы запросов неизбежны; они не должны приводить к двойному billing или двойному provisioning. |
| CR-OCS-018 | Стандарт должен включать correlation identifiers для трассировки действий между порталами, API, очередями и провайдерами. | Без сквозной трассировки невозможно расследовать cross-provider операции. |
| CR-OCS-019 | Стандарт должен поддерживать экспорт/импорт метаданных сервиса для переносимости между инсталляциями. | Уход от lock-in требует переносить не только данные, но и контекст сервиса. |
| CR-OCS-020 | Стандарт должен быть расширяемым без изменения ядра для каждого нового типа сервиса. | Экосистема должна расти быстрее ядра CloudRING. |

## Service Manifest

CloudRING должен иметь декларативный service manifest как источник истины.
Название формата может измениться, но назначение должно сохраниться.

| ID | Требование | Почему |
|---|---|---|
| CR-OCS-021 | Manifest должен описывать идентичность сервиса, namespace/tenant scope, окружения, зависимости, tasks, lifecycle APIs, UI, usage metrics и документацию. | В прототипе manifest оказался центральным контрактом между сервисом и платформой. |
| CR-OCS-022 | Manifest должен быть валидируемым до запуска сервиса. | Ошибки имени, зависимостей, задач и окружений должны находиться до runtime. |
| CR-OCS-023 | Manifest должен поддерживать глобальные настройки и environment-specific overrides. | Один сервис должен работать в local/staging/prod/private/edge без копирования конфигураций. |
| CR-OCS-024 | Manifest не должен хранить секреты в открытом виде. | Конфигурация и секреты имеют разный жизненный цикл и риск. |
| CR-OCS-025 | Manifest должен позволять объявить несколько компонентов одного типа с разными ролями. | Stateful-сервисы часто имеют master/replica/olap/cache/shard роли. |
| CR-OCS-026 | Manifest должен быть пригоден для генерации runtime-артефактов. | Производные файлы должны генерироваться платформой и быть одноразовыми. |
| CR-OCS-027 | Manifest должен быть читаемым человеком и машиной. | Он нужен разработчикам, операторам и AI-агентам. |

## Service Connector

| ID | Требование | Почему |
|---|---|---|
| CR-OCS-028 | Service Connector должен адаптировать внешний сервис к Open Cloud Standard без переписывания всего сервиса. | Это снижает барьер подключения ISV и existing services. |
| CR-OCS-029 | Connector должен явно объявлять, какие обязательные и опциональные capability он реализует. | Marketplace должен понимать уровень совместимости. |
| CR-OCS-030 | Connector должен передавать usage metrics, health, lifecycle events и error semantics в стандартизированном виде. | Платформа должна управлять разными сервисами одинаково. |
| CR-OCS-031 | Connector должен поддерживать sandbox/validation режим до публикации в marketplace. | Нельзя пускать непроверенный сервис в федерацию. |
| CR-OCS-032 | Connector должен иметь минимальные требования к безопасности, идентичности, secret handling и audit trail. | Connector становится границей доверия. |
