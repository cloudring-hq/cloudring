# Capability Evidence Matrix

## Назначение

Capability Evidence Matrix связывает detailed capability contracts с stage
readiness. Она нужна, чтобы AI-agent, человек и будущая команда проверяли не
только отдельный requirement, но и достаточность доказательств для конкретной
стадии CloudRING: local developer, private presence, provider, federation,
global network and self-evolving platform.

Matrix не выбирает implementation и не заменяет stage profiles. Она задает
минимальный proof shape, freshness, stop conditions and handoff expectations для
capabilities, которые проходят через несколько stages.

## Requirements

| ID | Требование | Почему | Acceptance |
|---|---|---|---|
| CR-CAPEVID-001 | Каждый detailed capability contract должен иметь stage-aware evidence mapping. | Capability может быть готова на Stage 2 и недостаточна на Stage 6. | Matrix links capability contract, stage, required evidence, freshness, owner and stop condition. |
| CR-CAPEVID-002 | Evidence mapping must distinguish proof, blocker, non-goal and future-stage dependency. | Agents must not turn future ambition into current-stage blocker. | Readiness report marks required-now, accepted non-goal, blocked and future-stage evidence separately. |
| CR-CAPEVID-003 | Infrastructure profiles must prove replaceable capability, capacity, health and exit by stage. | Infrastructure is the easiest place to hide lock-in. | Stage evidence includes profile version, state matrix, capacity, restore, local autonomy and backend deprecation/exit where relevant. |
| CR-CAPEVID-004 | Observability/support/operations must prove health, SLO/SLA, incident/support ownership and learning. | Operability is product readiness, not a dashboard screenshot. | Stage evidence includes health, incident, support, SLO/SLA, runbook, normalized audit report, recovery drill and learning records appropriate to scope. |
| CR-CAPEVID-005 | Portability/exit/cross-provider operations must prove export/import/restore and policy-safe movement. | Anti-lock-in is only real when exit and movement are tested. | Stage evidence includes portability profile, restore/import, preflight, residual-data closure and cross-provider validation where relevant. |
| CR-CAPEVID-006 | Self-service/agent operations must prove consequence-before-action and approval-safe automation. | One human with agents requires evidence, not hidden authority. | Stage evidence includes role journey, agent plan, risk class, approval, dry-run, validation, final evidence and manual-review bundle where needed. |
| CR-CAPEVID-007 | High-impact operations must include preflight and rollback/compensation evidence. | Data movement, billing and destructive lifecycle changes need stronger proof. | Readiness report links preflight, validation, rollback rehearsal or explicit irreversible warning with approval. |
| CR-CAPEVID-008 | Exit/offboarding evidence must include residual data, retention and closure status. | Provider, jurisdiction or technology exit can leave hidden obligations. | Exit/offboarding report shows retained data, deletion/retention windows, backups, evidence retention and closure status. |
| CR-CAPEVID-009 | Approval evidence must be scoped, current and revocable. | Broad/stale approval is not governance. | Approval artifact includes actor, approver, scope, risk, expiry/review, evidence, revocation path and dual-control where needed. |
| CR-CAPEVID-010 | Disconnected/private evidence must prove local control without global ownership. | Private and edge autonomy are core anti-lock-in promises. | Local readiness evidence shows health, IAM, policy, backup/export, incident/emergency actions and freshness/degraded sync state. |
| CR-CAPEVID-011 | Federation/global evidence must prove scoped exchange without central ownership takeover. | The global network must not become the new lock-in point. | Evidence shows participant owner, freshness, policy, trust, support/billing scope and local owner preservation. |
| CR-CAPEVID-012 | Evidence matrix must evolve from incidents, failed migrations and conformance drift. | The platform must not age into repeated toil. | Repeated failures create requirement, ADR, runbook, conformance update or explicit no-change rationale. |

| CR-CAPEVID-014 | Service Factory evidence must prove paved-road creation and local runtime readiness. | A scaffold is not a usable developer platform without local run, docs, tasks and validation. | Evidence includes template output, local preflight, dependency run, task result, docs preview, generated artifact inventory and candidate conformance. |
| CR-CAPEVID-015 | Security/supply-chain evidence must prove secret safety and artifact trust. | Open ecosystem requires proof before install, publish or run. | Evidence includes secret validation, brokered access audit, artifact provenance/integrity, dependency freshness, image hardening, sealed-image residue check, operational evidence redaction, release/freshness evidence, extension boundary and source-safety checks. |
| CR-CAPEVID-016 | Federation/global trust evidence must prove participant autonomy and scoped network governance. | Global network must not become central lock-in. | Evidence includes participant lifecycle, scoped sync, trust anchor state, conformance downgrade, dispute/suspension/offboarding and global discovery explanation. |
| CR-CAPEVID-017 | Manifest/schema evidence must prove field-level compatibility and secret-safe profiles. | Service contract cannot be reimplemented from vague manifest ideas. | Evidence includes required/optional/extension field matrix, environment precedence, secret-reference validation, field-level rule identity, timing/default/unknown-field behavior, deprecation policy and validation error catalog. |
| CR-CAPEVID-018 | Local runtime/task evidence must prove command parity and bounded execution. | Developer commands must be safe for humans, CI and agents. | Evidence includes runtime profile, preflight, command matrix, debug split, task runner boundary, structured results and local routing/conflict report. |
| CR-CAPEVID-019 | Documentation lifecycle evidence must prove support usefulness. | A running docs server is not enough for future support. | Evidence includes overview, API, architecture, runbook, FAQ, development flow, known limits, manifest linkage and docs check result. |
| CR-CAPEVID-020 | Plugin/UI extension evidence must prove trust boundary before execution or embed. | Extensions are executable and visual trust boundaries. | Evidence includes plugin manifest, permission/audit record, UI embed descriptor, lifecycle mount/update/unmount/failure cleanup, origin/host/isolation controls, scoped context, telemetry/redaction status, support owner and source-safety result. |
| CR-CAPEVID-021 | Usage gateway evidence must prove billable-event safety. | Billing trust fails if invalid usage disappears into logs. | Evidence includes versioned validation contract, payload-bound idempotency, duplicate/replay fixtures, decision ledger, period/metadata rules, queue/backpressure and shutdown/replay state, token-safe auth failure evidence, access freshness and shared state taxonomy. |
| CR-CAPEVID-022 | Source/history hygiene evidence must prove anonymized learning and remediation. | Legacy learning should not preserve private context or old secrets. | Evidence includes source-signal category, anonymization scan, secret-class discovery decision, rotation/remediation record and requirement/ADR/runbook update. |
| CR-CAPEVID-023 | Repeated-fix evidence must include regression or negative validation decision. | Fixes without coverage become recurring toil. | Evidence links repeated theme, threshold/window, failure mode and owner to regression/conformance/negative fixture or owner-approved no-test rationale. |
| CR-CAPEVID-024 | Source/history analysis must include coverage manifest. | Agents must not claim full audit from partial history analysis. | Evidence states current tree/ref/tag/deleted-path counts, generated/vendor exclusions, dirty-tree categories, sampling limits, source-safety classes, secret-safety handling, owner review decisions and anonymization result. |
| CR-CAPEVID-025 | Documentation decision-memory evidence must prove traceable product memory. | Docs, ADRs and source-pass lessons can otherwise become unowned prose or unsafe copies of old context. | Evidence links requirement, ADR/no-ADR rationale, docs package, template, example, scenario, conformance gate, source pass, owner, freshness, source-safety and explicit non-claims. |
| CR-CAPEVID-026 | Secret runtime readiness evidence must prove encrypted credential operation without leaking secrets or overclaiming readiness. | Secret-safe documentation and encryption-at-rest do not prove scoped runtime reconciliation, custody, rotation or fail-closed behavior. | Evidence links encrypted-data classification, scope binding, key custody, public certificate freshness, observed generation/conditions, install/delete retention, RBAC, health/metrics, rotation/degraded behavior, source-safety and explicit non-claims. |
| CR-CAPEVID-027 | Service dependency/deployment evidence must prove portable dependency model and generated artifact boundaries. | Local start success can hide ownerless dependencies, raw env values, generator limits and non-portable state. | Evidence links effective service model, dependency graph, component ownership, connection outputs, generated artifacts, env classification, conflict preflight, role matrix, portability, source-safety and non-claims. |
| CR-CAPEVID-028 | Base OS image factory evidence must prove reusable image readiness before private/provider promotion. | A built template can still contain residue, private profile data, mutable identity, missing guest readiness or hidden backend lock-in. | Evidence links image purpose, build input classification, install/provision summary, guest tooling, console/diagnostics, cleanup/sealing, immutable artifact identity, first-boot smoke, lifecycle, source-safety and non-claims. |
| CR-CAPEVID-029 | UI extension runtime certification evidence must prove embedded UI and validation readiness before private/provider publication. | Local preview, component tests or build success can hide host-authority, parity, accessibility, lifecycle cleanup, support and publication gaps. | Evidence links typed embed descriptor, host authority, scoped context, runtime lifecycle, validation phase/error contract, parity matrix, browser/runtime proof, accessibility/localization, artifact provenance, support owner, telemetry redaction, source-safety and non-claims. |
| CR-CAPEVID-030 | Settlement closure evidence must prove financial closeout before money is called settled. | Usage intake, invoice draft or release tag cannot prove reconciliation, dispute handling, participant shares or closeout export. | Evidence links closure run, input manifest, reconciliation, freeze gate, invoice/credit/refund trace, participant shares, dispute hold/release, party views, approval, export, retention, release/history, source-safety and non-claims. |

## Stage Evidence Matrix

| Stage | Infrastructure | Observability/Support | Portability/Exit | Self-Service/Agent |
|---|---|---|---|---|
| Stage 1 | Local profile, reference service portfolio evidence, developer workflow scenario evidence, presence bootstrap evidence, controlled automation evidence, effective service model, dependency graph and explicit non-production limits. | Service health, logs/metrics/traces or equivalent validation plus source-safe reference portfolio/workflow/bootstrap/dependency/env/task/plugin evidence. | Honest export story, dependency portability gaps and limitations for generated services, reference archetypes, tasks, plugins and workflow evidence confidence. | Developer flow, reference portfolio review, workflow negative cases, bootstrap dry-run, controlled task/plugin plan, agent plan, generated artifact review, dependency preflight and dry-run for risky local changes. |
| Stage 2 | Private profile, presence bootstrap activation, local autonomy, capacity, base OS image readiness, topology/endpoint ownership, backup/restore/failover and emergency control. | Local health, activation diagnostics, incident/audit, stateful audit bundle, secret runtime readiness, image cleanup/provenance, recovery/failover freshness, source-safe evidence and guided remediation. | Private backup/restore, export/import where applicable, activation rollback, image rollback/deprecation and local exit evidence. | Admin private presence activation flow, local agent operations, brokered secrets, base image evidence and source-safe operational bundles. |
| Stage 3 | Install target profile, service registry/catalog publication evidence, product service integration evidence, UI extension/runtime certification and unsupported/degraded states for private store. | Installed-service health/support, integration contract evidence, catalog publication audit, UI/runtime evidence and operational readiness. | Portability class visible before publication/install/update/remove, including integration, registry, UI extension dependency and validation limits. | Store publication, integration and lifecycle flow with blocked/manual-review bundle plus agent-readable registry, integration and UI certification evidence. |
| Stage 4 | Provider profile, tenant-facing capacity, portal experience evidence, release promotion evidence, support, billing runtime evidence, provider-local settlement closure and stateful readiness proof. | SLO/SLA, support owner, portal support handoff, support diagnostics package, support case/SLA/credit evidence, release/promotion health, incident/credit flow, maintenance, stateful audit blockers, recovery/failover freshness, closure blockers and source-safe diagnostics. | Provider exit, release rollback/retention, billing closeout export, support history and residual-data plan. | Provider onboarding, portal action approval, release promotion approval, tenant operations, support diagnostics approval, support case and credit approvals, billing/support/closure approvals and blocking audit remediation. |
| Stage 5 | Federation profile exchange, cross-cloud connectivity evidence and cross-participant settlement closure. | Cross-participant support handoff, scoped evidence sharing, dispute hold/release and participant-share views. | Cross-provider preflight, policy, compatibility, DR/migration validation and commercial closeout evidence. | Federation operation and settlement closure plans with approval and rollback/compensation. |
| Stage 6 | Global comparison of profiles without central owner takeover. | Global status, trust propagation and dispute-ready evidence. | Provider/jurisdiction/technology exit comparison and global evidence retention. | Global discovery/self-service with local ownership preserved. |
| Stage 7 | Profile drift and source/history gaps become requirement/ADR/runbook update. | Incidents/support toil become learning records with source-safe coverage manifest and documentation decision-memory evidence. | Failed migrations/exit issues update portability contract. | Agent lessons update requirements, approvals, conformance and runbooks without overclaiming source coverage. |

## Stop Conditions

Agent обязан остановиться и запросить owner/ADR/approval, если:

- capability is claimed ready without stage-aware evidence;
- readiness relies on a future-stage capability not declared as non-goal or ADR
  exception;
- evidence lacks owner, freshness, scope or validation;
- high-impact action lacks preflight, rollback/compensation or explicit
  irreversible warning;
- exit/offboarding leaves residual-data or evidence-retention state unknown;
- approval is stale, broad, ownerless or not revocable/reviewable;
- federation/global proof requires central ownership takeover;
- financial closure treats intake success, invoice draft or release tag as settlement proof;
- disputed, late, corrected or unknown usage settles without hold, approval or manual-review evidence;
- local/private readiness is claimed from binary install, config download or
  runtime start without bootstrap activation evidence;
- task/plugin/dependency/boilerplate readiness is claimed from local execution,
  generated files or raw command output without controlled automation evidence;
- service registry/catalog readiness is claimed from static assets, seed rows,
  local manifests, search output or debug/build success without publication
  evidence;
- developer workflow readiness is claimed from binary availability, docs,
  run profiles or showcase examples without workflow scenario evidence;
- release readiness is claimed from build success, tag, badge, CI entrypoint,
  manual job or local archive without immutable artifact, environment bundle,
  approval, rollback, retention and source-safe promotion evidence;
- shared-capability integration readiness is claimed from API docs, generated
  specs, credential existence, local tests or happy-path examples without
  product identity, scoped access, resource lifecycle, drift, fixtures, support
  handoff, decommission and source-safe integration evidence;
- support diagnostics readiness is claimed from scattered logs, generic health,
  generated docs, local/debug signals or raw export without read-only
  collection, lifecycle state, correlation, primary failure story, owner,
  retention, staged disclosure, source-safety and validation evidence;
- support case, SLA or credit readiness is claimed from free-form tickets,
  service docs, diagnostics, maintenance notes, usage intake or manual finance
  decisions without case binding, owner/escalation, SLA clock, credit policy,
  dispute hold/release, party-scoped views, correction lineage and approval
  evidence;
- portal or self-service UI readiness is claimed from blank frontend shell,
  documentation landing, local dev build, screenshot, component mount or
  navigation tree without role journeys, action parity, visible consequences,
  support-ready handoff, party-scoped views, release evidence and agent approval
  boundaries;
- repeated failure closes without learning decision;
- source/history analysis lacks coverage manifest;
- repeated fix cluster lacks regression/negative validation or no-test rationale.
