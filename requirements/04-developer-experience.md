# Developer Experience

Developer experience в CloudRING - это не удобство второго порядка.
Это основной способ масштабировать платформу через разработчиков, ISV,
интеграторов и AI-агентов.

## Локальная Среда

| ID | Требование | Почему |
|---|---|---|
| CR-DX-001 | Разработчик должен иметь возможность развернуть локальную среду CloudRING одной понятной процедурой. | Высокий порог входа убивает adoption. |
| CR-DX-002 | Локальная среда должна поддерживать зависимости сервиса: базы, object storage, secret store, messaging, observability. | Сервис редко работает в одиночестве. |
| CR-DX-003 | Локальная среда должна иметь унифицированный механизм start/stop/status. | Разработчик не должен помнить детали runtime-провайдера. |
| CR-DX-004 | Платформа должна поддерживать несколько runtime-provider backend и скрывать различия за общим интерфейсом. | Конкретные desktop/runtime технологии меняются и имеют ограничения. |
| CR-DX-005 | Локальный провайдер должен поддерживать ingress/multi-service сценарии. | Прототип показал, что локальная среда без нормального ingress ломает параллельную разработку. |
| CR-DX-006 | Платформа должна проверять минимальные ресурсы локальной среды до тяжелых операций. | Ошибки ресурсов должны быть понятными до запуска workload. |
| CR-DX-007 | Локальные производные файлы должны храниться в отдельном каталоге и не попадать в репозиторий сервиса. | Это снижает шум и предотвращает перенос локального состояния. |

## Шаблон Сервиса

| ID | Требование | Почему |
|---|---|---|
| CR-DX-008 | CloudRING должен предоставлять service template для каждого поддерживаемого типа сервиса. | Шаблон задает правильные defaults и снижает количество решений на старте. |
| CR-DX-009 | Шаблон должен включать единый entrypoint, структуру исходников, manifest, документацию, observability и тестовую базу. | Эти вещи должны быть нормой, а не ручной дисциплиной. |
| CR-DX-010 | Сервис не должен быть обязан хранить собственный build/deployment recipe, если платформа уже знает стандартный способ сборки. | Единая сборка снижает divergence между сервисами. |
| CR-DX-011 | Исключение для custom build/deployment должно быть явным архитектурным решением. | Гибкость нужна, но она должна быть осознанной. |
| CR-DX-012 | Имена сервисов должны валидироваться по ограничениям переносимости. | DNS, Kubernetes, URLs и federation налагают ограничения на имена. |
| CR-DX-013 | Шаблон должен требовать документацию сервиса как часть репозитория. | Сервис без документации невозможно поддерживать агентами. |
| CR-DX-014 | Шаблон должен включать health/readiness/metrics/tracing/logging conventions. | Observability должна появляться с первого коммита. |

## Документация

| ID | Требование | Почему |
|---|---|---|
| CR-DX-015 | Каждый сервис должен иметь документацию: overview, API, architecture, runbook, FAQ. | Эксплуатация и поддержка невозможны без понятного контекста. |
| CR-DX-016 | Платформа должна уметь запускать документацию сервиса локально. | Документацию нужно редактировать и проверять в dev loop. |
| CR-DX-017 | Документация должна быть связана с service manifest и lifecycle. | Документация не должна становиться забытым сайтом. |
| CR-DX-018 | Платформа должна поддерживать диаграммы и архитектурные схемы в документации. | AI-агентам и людям нужны визуальные связи между компонентами. |
| CR-DX-019 | Платформа должна иметь собственные docs для service developer, platform developer, operator и marketplace participant. | Разные роли решают разные задачи. |

## Плагины И Расширяемость

| ID | Требование | Почему |
|---|---|---|
| CR-DX-020 | CloudRING CLI/API должны расширяться через плагины для задач, которые не должны входить в ядро. | Специфичные процессы, enterprise-интеграции и закрытые автоматизации нельзя смешивать с OSS-ядром. |
| CR-DX-021 | Плагин не должен зависеть от одного языка реализации. | Экосистема шире одного стека. |
| CR-DX-022 | Плагин должен иметь строгий packaging contract, capability declaration и permission model. | Плагин является executable trust boundary. |
| CR-DX-023 | Плагины должны получать окружение сервиса через стандартный механизм, а не через ручные договоренности. | Это сохраняет совместимость с manifest и service lifecycle. |
| CR-DX-024 | Платформа должна рекомендовать task library для простых автоматизаций и плагины для сложных расширений. | Не каждую задачу нужно превращать в расширение ядра. |

## UI И Валидация

| ID | Требование | Почему |
|---|---|---|
| CR-DX-025 | Portal UI должен поддерживать встраиваемые UI-компоненты сервисов через typed context contract и host-controlled lifecycle. | Marketplace-сервисы должны приносить свой пользовательский опыт без поломки портала. |
| CR-DX-026 | UI-компоненты должны иметь явный bootstrap/provider contract для standalone development и embedded portal mode. | Портал должен контролировать контекст, зависимости, permissions, lifecycle и theming. |
| CR-DX-027 | Формы должны поддерживать declarative validation rules со stable rule id, structured code and понятными ошибками. | Ошибки ввода должны ловиться до вызова API без создания UI-only truth. |
| CR-DX-028 | Валидация должна различать reactive, blur, submit and state transitions вроде pristine, dirty, touched, pending, submitted and blocked. | Разные ошибки нужно показывать в разное время, чтобы UI не раздражал пользователя и агент видел тот же результат. |
| CR-DX-029 | UI-ошибки должны быть machine-readable и human-readable, включая code, field/path, severity, params and remediation. | AI-агентам нужны коды и контекст, людям нужны понятные сообщения. |
