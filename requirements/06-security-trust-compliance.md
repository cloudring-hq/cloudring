# Security, Trust And Compliance

CloudRING создается, чтобы уменьшить необходимость слепого доверия.
Безопасность здесь является продуктовой функцией, а не набором внутренних
практик.

## Trust Model

| ID | Требование | Почему |
|---|---|---|
| CR-SEC-001 | CloudRING должен проектироваться для недоверительной федерации участников. | Провайдеры, ISV и private-инсталляции не должны иметь неограниченного доверия друг к другу. |
| CR-SEC-002 | Все критичные действия должны иметь audit trail: кто, что, когда, от имени кого и почему сделал. | Без аудита невозможны расследования, compliance и AI-операции. |
| CR-SEC-003 | Платформа должна поддерживать проверяемую безопасность: политики, события, подписи, логи, артефакты и attestations. | "Trust me" не решает проблему cloud trust. |
| CR-SEC-004 | Права должны задаваться через identity, roles, scopes, policies и service accounts. | Глобальные токены и неограниченные права создают высокий риск. |
| CR-SEC-005 | Платформа должна поддерживать separation of duties для разработчика, оператора, провайдера, ISV, reseller и клиента. | Разные роли не должны иметь одинаковый доступ. |
| CR-SEC-006 | Платформа должна иметь понятную модель ownership для ресурсов, данных, секретов и billing events. | В федерации владение становится неоднозначным. |

## Secrets

| ID | Требование | Почему |
|---|---|---|
| CR-SEC-007 | Секреты не должны храниться в открытом виде в service manifest, docs, templates или generated files. | Источники показали риск утечки секретов в шаблонах и конфигурациях. |
| CR-SEC-008 | Платформа должна поддерживать GitOps-friendly encrypted secrets. | Инфраструктура должна быть воспроизводимой без раскрытия секретов в репозитории. |
| CR-SEC-009 | Расшифровка секретов должна происходить только в доверенной среде с контролируемым ключом. | Зашифрованный secret должен быть бесполезен вне целевого контекста. |
| CR-SEC-010 | Платформа должна поддерживать rotation, revocation и audit секретов. | Секреты устаревают и компрометируются. |
| CR-SEC-011 | Секреты должны иметь scopes по сервису, среде, tenant, provider и capability. | Секрет production не должен работать в dev или у другого провайдера. |
| CR-SEC-012 | Платформа должна явно различать конфигурацию и секреты. | Конфигурация часто публична внутри команды, секреты - нет. |

## Supply Chain

| ID | Требование | Почему |
|---|---|---|
| CR-SEC-013 | Все platform artifacts должны иметь происхождение, версию и проверку целостности. | Пользователь должен понимать, что именно установлено. |
| CR-SEC-014 | VM/container/runtime images должны собираться воспроизводимо через контролируемый pipeline. | Ручные образы невозможно надежно обновлять и проверять. |
| CR-SEC-015 | Образы должны проходить hardening, cleanup, удаление временных файлов и отключение небезопасных доступов. | Базовый образ является фундаментом доверия. |
| CR-SEC-016 | Сборочные секреты не должны попадать в финальные артефакты. | Supply chain часто ломается через утечки в build stage. |
| CR-SEC-017 | Платформа должна поддерживать vulnerability scanning и policy gates для marketplace-сервисов. | Marketplace без проверки переносит риск на клиентов. |
| CR-SEC-018 | Платформа должна поддерживать dependency provenance и обновления библиотек. | История прототипов показывает постоянные обновления зависимостей и общих библиотек. |

## Jurisdiction And Compliance

| ID | Требование | Почему |
|---|---|---|
| CR-SEC-019 | Пользователь должен выбирать или запрещать юрисдикции размещения данных и workload. | Jurisdiction lock является отдельной формой зависимости. |
| CR-SEC-020 | Платформа должна явно показывать, где находятся данные, кто их обрабатывает и какие провайдеры участвуют. | Пользователь должен принимать осознанное решение. |
| CR-SEC-021 | Платформа должна поддерживать policy enforcement для data residency. | Рекомендаций недостаточно, нужна автоматическая защита. |
| CR-SEC-022 | Платформа должна поддерживать compliance profiles для разных рынков и типов клиентов. | Enterprise/private клиенты живут в разных регуляторных режимах. |
| CR-SEC-023 | Cross-provider сценарии должны проверять совместимость policy всех участников. | DR/backup/replication может случайно вывести данные в запрещенную зону. |
| CR-SEC-024 | Платформа должна поддерживать privacy-preserving telemetry. | Мониторинг не должен раскрывать лишние пользовательские данные. |

## Error And Incident Transparency

| ID | Требование | Почему |
|---|---|---|
| CR-SEC-025 | Ошибки API должны иметь стабильные коды и безопасные сообщения. | Клиентам и агентам нужны machine-readable ошибки без утечки внутренностей. |
| CR-SEC-026 | Security events должны быть доступны владельцу ресурса и уполномоченным операторам. | Владельцу нужен контроль над своей инфраструктурой. |
| CR-SEC-027 | Платформа должна поддерживать incident timeline с correlation IDs. | Инциденты в распределенной системе нельзя расследовать по отдельным логам. |
