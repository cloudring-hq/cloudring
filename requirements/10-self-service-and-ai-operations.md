# Self-Service And AI Operations

CloudRING должен быть полностью self-service со стороны пользователя,
администратора, провайдера, разработчика и AI-агента.

Идеальный опыт: сложная облачная операция ощущается как ясный продуктовый поток.
Пользователь понимает, что произойдет, сколько это стоит, где это будет
работать, какие риски есть и как откатиться.

## Пользовательский Self-Service

| ID | Требование | Почему |
|---|---|---|
| CR-SELF-001 | Пользователь должен самостоятельно найти, сравнить, заказать, изменить, остановить и удалить сервис. | Облако без self-service превращается в managed services queue. |
| CR-SELF-002 | Выбор сервиса должен показывать цену, регион, jurisdiction, SLA, provider, dependencies и trust/compliance profile. | Пользователь покупает не только функцию, но и контекст риска. |
| CR-SELF-003 | Пользователь должен видеть текущее потребление, прогноз стоимости и billing events. | Непрозрачный счет создает недоверие. |
| CR-SELF-004 | Пользователь должен видеть состояние сервиса и понятные действия восстановления. | Self-service должен работать и в проблемных состояниях. |
| CR-SELF-005 | Пользователь должен экспортировать данные и метаданные сервиса там, где сервисный контракт это допускает. | Уход от lock-in должен быть не лозунгом, а кнопкой/процедурой. |
| CR-SELF-006 | Пользователь должен видеть, какие участники federation задействованы в его сервисе. | Cross-provider прозрачность нужна для доверия. |

## Admin Self-Service

| ID | Требование | Почему |
|---|---|---|
| CR-SELF-007 | Администратор должен самостоятельно развернуть CloudRING-инсталляцию из supported installer/distribution. | Платформа должна масштабироваться без консалтинга на каждом шаге. |
| CR-SELF-008 | Администратор должен видеть health всей инсталляции, capacity, alerts, upgrade status и policy violations. | Оператору нужна единая картина. |
| CR-SELF-009 | Администратор должен запускать upgrades через plan/apply/validate flow. | Обновления должны быть предсказуемыми. |
| CR-SELF-010 | Администратор должен иметь guided remediation для типовых проблем. | Один человек с агентами должен поддерживать платформу. |
| CR-SELF-011 | Администратор должен подключать identity, network, storage, backup и monitoring providers через wizard/API. | Интеграция инфраструктуры должна быть продуктовой функцией. |
| CR-SELF-012 | Администратор должен управлять policy: allowed regions, providers, service classes, budgets, data residency, security baselines. | Управление облаком - это управление ограничениями. |

## Provider Self-Service

| ID | Требование | Почему |
|---|---|---|
| CR-SELF-013 | Провайдер должен самостоятельно подключить presence к CloudRING federation через certification flow. | Глобальная сеть не может масштабироваться ручным onboarding. |
| CR-SELF-014 | Провайдер должен публиковать capacity, regions, prices, SLA и service availability. | Marketplace должен иметь актуальные предложения. |
| CR-SELF-015 | Провайдер должен видеть settlement, revenue share, usage, disputes и quality metrics. | Экономика federation должна быть прозрачной. |
| CR-SELF-016 | Провайдер должен иметь инструменты для публикации собственных сервисов в сеть. | CloudRING должен стимулировать инновации провайдеров. |

## ISV / Developer Self-Service

| ID | Требование | Почему |
|---|---|---|
| CR-SELF-017 | Разработчик должен создать сервис из шаблона, запустить зависимости, проверить контракт и опубликовать candidate build без ручных согласований. | Marketplace должен расти за счет низкого барьера входа. |
| CR-SELF-018 | ISV должен видеть conformance, security, billing и documentation readiness до публикации. | Сертификация должна быть понятной, а не бюрократической. |
| CR-SELF-019 | ISV должен задавать коммерческие планы, поддерживаемые среды и ограничения использования. | Сервис является продуктом, а не только кодом. |
| CR-SELF-020 | ISV должен получать usage/revenue/support feedback по своим сервисам. | Разработчик должен улучшать сервис на основе данных. |

## AI-Agent Operations

| ID | Требование | Почему |
|---|---|---|
| CR-SELF-021 | AI-агент должен иметь отдельную identity, scopes и audit trail. | Агент не должен действовать как безымянный root. |
| CR-SELF-022 | Любая агентская операция должна быть описана как goal, plan, actions, validations, rollback/compensation. | Это делает автономную работу проверяемой. |
| CR-SELF-023 | Агент должен уметь читать requirements, ADR, runbooks, telemetry и source maps перед действием. | Иначе агент повторит старые ошибки. |
| CR-SELF-024 | Агент должен отличать read-only investigation, safe change, risky change и destructive change. | Риск должен управляться явно. |
| CR-SELF-025 | Агент должен иметь dry-run/simulation mode для инфраструктурных изменений. | Без предварительной проверки автономность опасна. |
| CR-SELF-026 | Агент должен оставлять итоговую запись: что изменено, почему, какие проверки пройдены, какие риски остались. | Человеку нужно доверять не словам, а следу действий. |
| CR-SELF-027 | Агент должен уметь создавать новое requirement/ADR при обнаружении нового операционного урока. | Платформа должна учиться на опыте. |
| CR-SELF-028 | Агент не должен иметь доступ к секретам напрямую, если может выполнить задачу через brokered capability. | Минимизация секретов снижает blast radius. |

## Self-Service Критерии Качества

Каждый self-service поток должен отвечать на вопросы:

1. Что будет создано или изменено?
2. Где это будет работать?
3. Кто отвечает за результат?
4. Сколько это стоит сейчас и может стоить позже?
5. Какие данные и секреты будут затронуты?
6. Какие policy применяются?
7. Как проверить успех?
8. Как остановить, откатить или перенести?
9. Что увидит пользователь при деградации?
10. Может ли этот же поток выполнить AI-агент безопасно?
