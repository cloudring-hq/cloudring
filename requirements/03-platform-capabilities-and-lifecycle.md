# Platform Capabilities And Lifecycle

Этот документ описывает, какие capability должна предоставлять CloudRING как
платформа.
Здесь не фиксируется конкретная реализация.

## Базовые Capability

| ID | Требование | Почему |
|---|---|---|
| CR-PLAT-001 | CloudRING должен предоставлять IAM и Resource Manager как базовые capability любой инсталляции. | Все остальные сервисы зависят от идентичности, прав и учета ресурсов. |
| CR-PLAT-002 | CloudRING должен поддерживать compute services через заменяемые backend-реализации. | Compute нужен в public, private, edge и single-host сценариях. |
| CR-PLAT-003 | CloudRING должен поддерживать network services: адресация, маршрутизация, ingress, service discovery, cross-cloud connect. | Сервисы должны быть доступны и связаны между средами. |
| CR-PLAT-004 | CloudRING должен поддерживать storage services: block, object, backup storage, snapshots, replication. | Stateful и enterprise-сценарии невозможны без storage. |
| CR-PLAT-005 | CloudRING должен поддерживать container platform как один из способов запуска workload. | Container lifecycle нужен разработчикам и сервисам, но не должен быть единственным способом. |
| CR-PLAT-006 | CloudRING должен поддерживать VM/bare metal lifecycle для private и enterprise workloads. | Не все workloads могут или должны быть контейнеризированы. |
| CR-PLAT-007 | CloudRING должен поддерживать public cloud connector capability. | Федерация должна уметь использовать внешние presence и расширять покрытие. |
| CR-PLAT-008 | CloudRING должен поддерживать marketplace services как первый класс, а не как внешний каталог ссылок. | Услуга должна проходить lifecycle внутри платформы. |
| CR-PLAT-009 | CloudRING должен поддерживать development services: шаблоны, SDK, локальные runtime, тестовые сервисы, документацию, task library. | Разработчики и AI-агенты являются производителями платформенной ценности. |
| CR-PLAT-010 | CloudRING должен поддерживать documentation services для платформы и каждого сервиса. | Знание должно быть частью продукта, а не отдельно живущим wiki. |

## Lifecycle Сервиса

| ID | Требование | Почему |
|---|---|---|
| CR-PLAT-011 | Сервис должен проходить lifecycle от разработки до marketplace через один набор контрактов. | Разрыв между dev и prod создает ошибки и снижает скорость. |
| CR-PLAT-012 | Платформа должна поддерживать create/init сервиса из шаблона. | Шаблон снижает порог входа и задает стандарты наблюдаемости, документации и сборки. |
| CR-PLAT-013 | Платформа должна поддерживать local debug зависимостей сервиса без обязательного запуска самого сервиса внутри платформы. | Разработчику нужен обычный отладчик и быстрый feedback loop. |
| CR-PLAT-014 | Платформа должна поддерживать service start/stop/status/logs как зрелые lifecycle-команды. | Эксплуатация требует прозрачного управления состоянием. |
| CR-PLAT-015 | Незрелые команды lifecycle должны явно маркироваться и не маскироваться под production-ready. | История прототипа показала риск нестабильных команд. |
| CR-PLAT-016 | Платформа должна уметь генерировать runtime-спеки для разных targets из одного service manifest. | Один контракт должен работать локально, в private cloud, public cloud и edge. |
| CR-PLAT-017 | Платформа должна хранить производные runtime-файлы отдельно от исходников сервиса. | Это уменьшает churn и предотвращает дрейф инфраструктурной механики. |
| CR-PLAT-018 | Платформа должна поддерживать clean teardown локальных и тестовых зависимостей сервиса. | Разработчик должен легко возвращаться к чистому состоянию. |
| CR-PLAT-019 | Платформа должна поддерживать discovery адресов и credentials зависимостей через стандартизированный environment/export/API. | Ручной поиск портов и адресов ломает developer experience. |
| CR-PLAT-020 | Платформа должна поддерживать несколько одновременно развернутых сервисов в локальной среде. | Микросервисная разработка часто требует интеграционных сценариев. |
| CR-PLAT-021 | Платформа должна защищать от конфликтов имен, портов, namespace и компонентов. | Локальный и shared runtime быстро ломаются из-за конфликтов. |
| CR-PLAT-022 | Платформа должна поддерживать service docs preview/edit lifecycle. | Документация сервиса должна проверяться так же, как код. |

## Компоненты Сервиса

| ID | Требование | Почему |
|---|---|---|
| CR-PLAT-023 | Сервис должен объявлять необходимые компоненты через контракт, а не через ручные инструкции. | Платформа должна автоматически готовить зависимости. |
| CR-PLAT-024 | Компоненты должны иметь тип, имя, роль, статус enabled/disabled и connection contract. | Один тип компонента может использоваться несколько раз в разных ролях. |
| CR-PLAT-025 | Платформенные компоненты и сервисные компоненты должны различаться. | Общие компоненты вроде messaging/observability обслуживают многие сервисы, а service-owned компоненты принадлежат конкретному сервису. |
| CR-PLAT-026 | Каждый компонент должен публиковать machine-readable connection details без раскрытия секретов в открытом виде. | Автоматизация должна подключаться безопасно. |
| CR-PLAT-027 | Компоненты должны поддерживать compatibility profiles для local, staging, production, private и edge. | Один и тот же компонент не всегда одинаков в разных средах. |
| CR-PLAT-028 | Stateful-компоненты должны иметь lifecycle для backup, restore, migration, upgrade и failover. | Stateful-сервисы требуют больше, чем "запустить контейнер". |

## Task Library

| ID | Требование | Почему |
|---|---|---|
| CR-PLAT-029 | Платформа должна иметь библиотеку переносимых задач, объявляемых сервисом. | Это заменяет уникальные Makefile-практики единым подходом. |
| CR-PLAT-030 | Одна и та же задача должна запускаться локально и в CI с одинаковым контрактом. | Совместимость инструментов снижает порог входа и количество ошибок. |
| CR-PLAT-031 | Задачи должны выполняться в изолированной среде с контролируемыми входами. | Разные машины разработчиков не должны менять результат. |
| CR-PLAT-032 | Платформа должна поддерживать стандартные задачи: lint, test, build, db migrate, db seed, docs, security scan, compatibility check. | Эти операции повторяются во всех сервисах. |
| CR-PLAT-033 | Платформа должна позволять сервису объявлять custom tasks, если они следуют контракту. | Не все автоматизации можно предсказать заранее. |
| CR-PLAT-034 | Платформа должна запрещать или явно обрабатывать дубликаты имен задач. | Неоднозначность задач ломает автоматизацию. |
